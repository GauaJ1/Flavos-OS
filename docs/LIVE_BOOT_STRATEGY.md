# Flavos OS - Live Boot Strategy

This document outlines the architectural choices, optimizations, and security posture for the Flavos OS Live Environment. This strategy enforces strict performance guidelines (especially for systems with only 2GB of RAM) and security boundaries.

## 1. Architectural Approach

**Choice:** Custom Pipeline via `live-boot`
Instead of using external toolchains like `live-build`, we will reuse the existing Flavos OS `rootfs` generated by `01-create-rootfs.sh`. 

**Pipeline Flow:**
1. A base `rootfs` is prepared.
2. The `live-boot` and `live-boot-initramfs-tools` packages are injected into the rootfs.
3. The rootfs is compressed into a read-only `squashfs`.
4. A Hybrid ISO is generated using `xorriso` and GRUB, packaging the kernel, initramfs, and squashfs image.

## 2. Boot Parameters & Memory Optimization

Systems with 2GB of RAM (like legacy LGA 775 setups) are extremely sensitive to `tmpfs` consumption. The `overlayfs` used by live systems stores all session modifications in RAM.

### The `toram` Parameter
- **Status:** **PROHIBITED**
- **Reasoning:** The `toram` option copies the entire filesystem into RAM. For a system with only 2GB, this immediately starves the OS of memory, leading to out-of-memory (OOM) killer invocation upon opening a browser. The squashfs must be read directly from the boot medium (USB/CD).

### Default Boot Parameters
To ensure stability and memory conservation, the default boot entry will utilize:
```text
boot=live components nopersistence noeject overlay-size=512M
```
- `nopersistence`: Ensures the system is entirely amnesic. No state is saved to disk.
- `overlay-size=512M`: Strictly caps the `tmpfs` upperdir for overlayfs to 512 MB, protecting the system from exhausting its 2GB total RAM.

### Boot Menu Entries
1. **Flavos OS Live**
   *Parameters:* `boot=live components nopersistence noeject overlay-size=512M`
2. **Flavos OS Live (safe graphics)**
   *Parameters:* `boot=live components nopersistence noeject overlay-size=512M nomodeset`
   *Purpose:* Fallback for older hardware with failing or incompatible GPU drivers.
3. **Flavos OS Live (low RAM)**
   *Parameters:* `boot=live components nopersistence noeject overlay-size=384M`
   *Purpose:* For environments with shared VRAM or extreme RAM constraints, reducing the overlay cap to 384 MB.

## 3. Storage & Compression

### Primary Format
**Zstd (Zstandard) - Moderate Compression**
- **Command:** `mksquashfs rootfs/ filesystem.squashfs -comp zstd -Xcompression-level 3 -b 256K`
- **Reasoning:** Zstd provides an excellent balance between decompression speed and ratio. Level 3 is fast enough for older CPUs while still keeping the file size small enough to reduce USB 2.0 I/O bottleneck overhead. The 256K block size keeps memory usage reasonable during extraction.

### Alternative (For Extremely Weak CPUs)
**LZ4 Compression**
- If the target CPU struggles significantly with Zstd decompression (causing CPU bottlenecking over I/O), `lz4` should be evaluated. It trades a larger file size for blazing-fast decompression speeds.

## 4. Boot Compatibility
The generated Hybrid ISO must support:
- **Legacy BIOS:** via `grub-pc-bin`
- **UEFI (64-bit):** via `grub-efi-amd64-bin`

*Note: 32-bit UEFI (`grub-efi-ia32-bin`) is excluded from the initial scope to keep the toolchain minimal, unless a specific hardware target requires it.*

## 5. Security & Persistence

### Amnesic by Default
The initial prototype and subsequent V1 release will be strictly **amnesic**. `nopersistence` is hardcoded in the boot parameters.

- **Threat Model Mitigation:** Any malware or misconfiguration introduced during a session is wiped upon reboot.
- **Data Integrity:** Users must mount an external drive explicitly to save data, preventing accidental data loss in the volatile `tmpfs`.
- **Future Scope:** Persistent partitions (ext4 labeled `persistence`) may be supported in a future stage but remain disabled for the prototype.
